## Context

В настоящее время приложение имеет два разных экрана для управления колясками (профилями): `ProfilePassportScreen` (в основном используется для создания новых) и блок паспорта внутри `SettingsScreen` (для редактирования текущего профиля). Такое дублирование вызывает путаницу. Кроме того, отсутствует механизм удаления профилей, что оставляет в БД "мусор" после тестов.

## Goals / Non-Goals

**Goals:**
- Удалить `ProfilePassportScreen.kt` из навигации и кодовой базы.
- Реализовать автосоздание профиля при нажатии `+` (включая логику инкремента: "Новая коляска", "Новая коляска 2" и т.д.).
- Добавить возможность удаления профиля (коляски) прямо из `SettingsScreen`.
- Внедрить диалог разрешения конфликтов при удалении коляски с историей (наследованные треки): удаление вместе с историей или перенос истории на другую коляску.

**Non-Goals:**
- Массовое редактирование профилей из одного экрана менеджера (каждая редактируется индивидуально).
- Восстановление удалённых профилей из корзины (undo).

## Decisions

**Решение 1: Удаление ProfilePassportScreen**
Мы полностью удаляем этот компонент из UI и его ViewModel (`ProfilePassportViewModel`). Вся логика управления активным профилем "Паспорта" остается в `SettingsViewModel`. Это уменьшает размер бандла и количество архитектурных связей.

**Решение 2: Автосоздание ("Новая коляска")**
Вместо того, чтобы открывать пустой экран сохранения, `MainViewModel.createAndSwitchProfile()` будет создавать запись в Room с базовым именем из ресурсов. Если имя конфликтует с существующими — добавляется суффикс (2, 3 и т.д.). После этого происходит переход на `Routes.SETTINGS`, где пользователь может изменить имя.

**Решение 3: Диалог удаления (Safe Delete) с каскадным обновлением**
В `SettingsScreen` будет добавлена кнопка "Удалить коляску" (скрыта, если Profiles.size == 1). При удалении профиля мы сначала проверяем количество треков (`DaySessionEntity`) для этого профиля:
* Если их 0: Происходит удаление, и переключение активного профиля на любой первый доступный из БД.
* Если > 0: Выводим диалог AlertDialog, в котором дается выбор (RadioButtons): удалить коляску со всей историей ИЛИ сохранить историю, переназначив на другую (выбирается из ExposedDropdownMenu). Релокация осуществляется mass-update запросом `UPDATE DaySessionEntity SET profileId = :newId WHERE profileId = :oldId`, что очень быстро выполняется на уровне SQLite.

## Risks / Trade-offs

- [Risk] Переназначение треков на коляску другого типа (с Электрической на Ручную) может сделать метрики LED-зарядов неактуальными. → [Mitigation] Принимается как есть. Мы предоставляем инструмент для "спасения".
- [Risk] Одиночный профиль-зомби (создали и забыли). → [Mitigation] Оставляем в интерфейсе механизм легкого переключения; если профиль пустой, он не мозолит глаза.
