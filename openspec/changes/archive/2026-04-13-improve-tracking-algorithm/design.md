## Context

В приложении существует модуль трекинга (`TrackingService`), который фиксирует координаты и классифицирует их (ходьба на коляске или транспорт) на основе порога скорости `location.speed > maxSpeed * 1.2`. Тестирование показало критические недостатки этого подхода: точные устройства в транспорте могут не пересекать порог в условиях пробок, а неточные устройства (или при холодном старте) могут фиксировать ложные скачки («звезда»). Дополнительно обнаружены проблемы со слишком длинными интервалами без обновления GPS (когда MIUI убивает процесс).
В UI истории есть проблемы с отображением прерывистой линии «Транспорт» и группировкой нескольких профилей за день.

## Goals / Non-Goals

**Goals:**
- Внедрить надежное определение транспорта с помощью Android `ActivityRecognition API`.
- Реализовать фильтрацию начальных «прыжков» GPS (проверка `accuracy` < 30 м) и отслеживание потери сигнала (разрыв линии при тайм-ауте).
- Починить баг с `profile_id=1` при создании нового `TrackSegment` в базе.
- Создать в слое Repository новый метод для выборки всех сессий за день (независимо от профиля коляски) и добавить тумблер в UI для этого.
- Пофиксить разбиение и отрисовку транспортных участков в `HistoryMapScreen`.

**Non-Goals:**
- Не переписываем архитектуру Repository и базы данных с нуля; работаем в рамках текущих таблиц и добавляем нужные JOIN.
- Не интегрируем машинное обучение для транспорта (полагаемся на встроенный Google Play Services).
- Не используем Wakelocks (пока), ограничимся фильтрацией последствий убийства процесса.

## Decisions

1. **Activity Recognition API Integration:**
   - Добавить библиотеку Play Services `play-services-location` для `ActivityRecognitionClient`.
   - Если статус `IN_VEHICLE` подтвержден, `TrackingService` помечает следующие точки как транспортные (сохраняя также гистерезис).
   - Запрашивать разрешение `ACTIVITY_RECOGNITION` у Android 10+ пользователей перед стартом трека.

2. **GPS Filtering & Segment Management Rules:**
   - Игнорировать любые точки с `accuracy > 30` во время первых 30 секунд старта сессии.
   - Если `current_time - last_location_time > 120_000` (2 минуты), создавать новый `TrackSegment`, чтобы `Polyline` на карте обрывался и не пересекал дома напрямую. (Если MIUI убьет и перезапустит сервис — `TrackSegment` всё равно будет новым).

3. **Database structure tweak:**
   - Внутри `StartTrackUseCase` гарантированно передавать и сохранять `profile_id` текущей сессии свича в `track_segment`.

4. **History UI "All Profiles" Toggle:**
   - В `HistoryViewModel` добавляем State `showAllProfiles: Boolean`.
   - При `false` запрашиваем сессии только для активного `profile_id`.
   - При `true` запрашиваем вообще все сессии, группируя их исключительно по `date` (через новый SQL-запрос `getAllSessionsGroupedByDate`). В UI карточки показываем иконки колясок, входивших в этот день.

5. **Map Visuals (Red Dashed Line):**
   - Убедимся, что `DrawSegment` функция корректно получает точки одного цвета и строит массив Path, вместо того чтобы строить одну прерывистую линию, которая ломается. Рисовать транспорт `pathEffect = PathEffect.dashPathEffect`.

## Risks / Trade-offs

- Activity Recognition API работает не моментально (задержка около 30-40 секунд на распознавание). Избежать ложноположительных в этот момент поможет старый fallback на гистерезис `location.speed`.
- Дополнительный запрос разрешения усложнит онбординг пользователя (нужно вывешивать отдельный диалог).
- Обновление таблиц и миграция: баг в `track_segment.profile_id=1` исправить мы сможем только для новых записей, старые останутся как есть.
