## Context

Приложение FigaGo использует `FusedLocationProviderClient` для получения GPS координат. Для записи трека на длительных дистанциях критично потребление заряда батареи. Операционная система Android (особенно в Doze Mode) может агрессивно ограничивать фоновую активность. Кроме того, системы OEM (Xiaomi, Huawei, Samsung и др.) требуют наличия специфичных разрешений (исключение из энергосбережения), без которых Foreground Service с локацией убивается системой через несколько минут работы в фоне. 

Вторая критическая проблема: архитектура хранения Data Layer. В данный момент `DaySessionEntity` (поездки) не содержит `profileId`, из-за чего вкладка истории отображает треки всех колясок вперемешку с дефолтными иконками. Кроме того, UX экрана Истории требует доработки (удаление дублирующих элементов навигации). 

Требуется внедрить систему предстартовых проверок, механизм динамического пакетирования точек геолокации, а также рефакторинг сохранения сессий с привязкой к активному профилю.

## Goals / Non-Goals

**Goals:**
- Реализовать предстартовую проверку системных разрешений с помощью интерактивного экрана самодиагностики.
- Динамически переключать `maxUpdateDelayMillis` в `LocationRequest` при переходах жизненного цикла (onResume / onPause).
- Обеспечить атомарное и быстрое сохранение батча полученных точек в Room-БД.
- **Привязать все новые сессии к `profileId` и отфильтровать отображение экрана истории по текущей активной коляске**.
- Улучшить UI экрана истории (убрать TopAppBar, вынести кнопку полного удаления вниз списка).

**Non-Goals:**
- Интеграция с закрытыми OEM API (автозапуск будет использовать UI-компоненты или стандартные Intent для общих настроек, fallback на 'Детали приложения').
- Миграция старых данных истории (старые треки без profileId могут быть либо привязаны к дефолтному профилю, либо показаны для всех профилей - по умолчанию `profileId` можно сделать nullable или привязать к первому существующему профилю).

## Decisions

**1. Механизм переключения Batching Mode**
   - *Решение*: Использовать `LifecycleObserver` на экране трекинга, отправляющий команду в `TrackingService` (Foreground UI `setMaxUpdateDelayMillis(0)`, Background UI `setMaxUpdateDelayMillis(30000)`). Обновление происходит через `removeLocationUpdates` и повторный `requestLocationUpdates()`.

**2. Сохранение батчей в Room**
   - *Решение*: Использовать `@Insert` для списка данных внутри `@Transaction`. 

**3. Изоляция сессий по профилям (Profile Isolation)**
   - *Решение*: Модифицировать `DaySessionEntity`, добавив колонку `profileId` (Reference на `ProfileEntity`). В `SessionRepository` перед созданием сессии запрашивать активный профиль. Обновить `GetHistoryUseCase` так, чтобы он возвращал `Flow<List<DaySession>>`, отфильтрованный по `activeProfileId`.

**4. Рефакторинг UX Истории (History Screen)**
   - *Решение*: Удалить `TopAppBar` из метода `Scaffold` в `HistoryScreen.kt`. Кнопку "Очистить всю историю" перенести в подвал списка `LazyColumn` для предотвращения случайных нажатий пользователем.

**5. Выбор иконки профиля (Wheelchair Icon Picker)**
   - *Решение*: Разработать UI-компонент (например, `ExposedDropdownMenuBox` или кастомное диалоговое окно / сетку `LazyVerticalGrid`) на экране `ProfilePassportScreen`. Компонент будет отображать список доступных Android Resource ID (`R.drawable.xxx`), включая добавленные SVG-векторы по моделям (Permobil F, Permobil M, Optimus, etc.).
   - *Обоснование*: `ProfileEntity` УЖЕ хранит `iconId` как `Int`. Изменение базы данных не требуется. Достаточно добавить новые ресурсы в `res/drawable` формата XML (VectorDrawable) и связать их с выпадающим списком.

**6. Дашборд и точное форматирование (Distance Formatting)**
   - *Решение*: В `DashboardScreen` поменять местами метрики: "Пробег" переместить в центр экрана, а "Текущую скорость" убрать или заменить на среднюю/время наверху левее-правее.
   - *Решение для форматирования UI*: Обновить `FormatMetricsUseCase`, чтобы он возвращал расстояние с учетом размерности. Если < 1км -> в метрах (целое число), если > 1км -> километры и метры. Учесть метрическую и имперскую системы.
   - *Решение для TTS*: Обновить функцию `buildAnnouncementText` в `TtsAnnouncerService`. При пробеге < 1000 метров произносить "пробег 250 метров". При $>1000$ произносить "1 километр 200 метров" (вычисляется делением по модулю).

## Risks / Trade-offs

- **Risk**: Старые устройства могут не поддерживать аппаратный batching.
  - **Mitigation**: GMS `FusedLocationProviderClient` автоматически переходит на fallback-эмуляцию.
- **Risk**: Миграция Room DB `DaySessionEntity` может привести к потере данных, если не указать `defaultValue` или не написать скрипт миграции.
  - **Mitigation**: Укажем `defaultValue` для новой колонки или реализуем простую миграцию, где старым сессиям присваивается ID профиля по-умолчанию (или null).
- **Risk**: Фрагментация Intent-ов настроек на разных устройствах.
  - **Mitigation**: Использовать `Settings.ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`, fallback на 'Детали приложения'.
